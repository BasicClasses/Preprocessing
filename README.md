# Препроцессинг

## Этапы компиляции

Четыре основных этапа

- Препроцессирование
- Компиляция (трансляция в ассемблер)
- Ассемблирование
- Компоновка (Линковка)

### Препроцессирование

Это отдельная программа, которая подготавливает текст для компилятора. Препроцессор работает с директивами — это все строки, начинающиеся с символа # 
Что там происходит 
- Удаление комментариев: Все комментарии (// и /* */) заменяются пробелами. Компилятор их не увидит.
- Обработка директив #include: Препроцессор берет содержимое файла stdio.h (или любого другого) и буквально вставляет его в текст вашей программы вместо строки #include.
  Именно поэтому наша программа "толстеет" еще до реальной компиляции.
- Макроподстановки (#define): Препроцессор проходит по коду и заменяет все макросы (например, #define PI 3.14) на их значения.
- Условная компиляция: Обрабатываются директивы #ifdef, #ifndef, #endif. Если условие ложно, куски кода просто выбрасываются из текста.

### Компиляция

Берется "чистый" файл без директив и переводит его на язык ассемблера.

- Лексический анализ: Исходный код разбивается на токены (ключевые слова int, идентификаторы main, операторы =, литералы).
- Синтаксический анализ (Парсинг): Из последовательности токенов строится синтаксическое дерево (AST — Abstract Syntax Tree).
  На этом этапе проверяется, правильно ли расставлены точки с запятой и скобки.
- Семантический анализ: Проверка смысла. Например, компилятор проверяет, не пытаетесь ли вы сложить указатель со строкой или вызвать функцию, которую не объявили.
- Оптимизация: Компилятор пытается сделать код быстрее или короче. Например, если вы написали i = 2 + 3;, оптимизатор может сразу заменить это на i = 5;,
  чтобы программа не тратила время на сложение констант.
- Генерация кода: На выходе создается файл с расширением .s или .asm, содержащий инструкции на языке ассемблера.

### Ассемблирование
Ассемблер — это транслятор с языка символьных инструкций в настоящий машинный код — последовательность байт, понятную процессору.

### Компоновка / Линковка

Финальный этап выполняет Линкер. 

- Связать все объектные файлы: Если наша программа состоит из нескольких .c файлов (например, main.c и utils.c), линкер соберет их объектные файлы (main.o, utils.o) вместе.
- Связать с библиотеками: Линкер находит код библиотечных функций и либо встраивает его в нашу программу (если библиотека статическая, .a),
  либо прописывает ссылки на системные библиотеки (если библиотека динамическая, .so или .dll).
- Разрешение символов (Relocation): Линкер подставляет реальные адреса. Он берет вызов функции printf в нашем коде и заменяет его
  инструкцией перехода на конкретный адрес в памяти, где эта функция находится.
- Построение окончательного образа: Создается исполняемый файл (по умолчанию a.out или program.exe),
  у которого есть заголовок, сегменты кода (text), данных (data) и таблицы для загрузчика операционной системы.

## Директивы препроцессора

Препроцессор — это отдельная программа, которая обрабатывает исходный код ещё до того, как он попадёт в компилятор. 
Все команды препроцессора начинаются с символа # и называются директивами. Они не являются частью языка, но существенно 
расширяют его возможности: позволяют включать файлы, определять константы, создавать макросы и управлять условной компиляцией. 
Рассмотрим основные директивы: #include, #define, условные директивы (#if, #ifdef и др.), а также некоторые специальные возможности. 

### Директива #include — включение файлов

Формы записи: 
- #include <имя_файла> — для системных заголовочных файлов (поиск в стандартных каталогах).
- #include "имя_файла" — для пользовательских файлов (поиск начинается в текущей папке).

Пример:
```
#include <stdio.h>      // подключаем стандартный ввод-вывод
#include "my_lib.h"     // подключаем собственную библиотеку
```
**Важно:** включение одного и того же файла много раз может привести к повторному определению. 
Для защиты от этого используют заголовочные стражи.

### Директива #define — макроподстановки
Позволяет задать идентификатор, который будет заменён на заданную последовательность символов. 
Бывает двух видов: макросы без аргументов (константы) и макросы с аргументами (похожи на функции). 

Простые макросы (константы)
```
#define PI 3.14159
#define MAX_SIZE 100
```
Макросы с аргументами
```
define SQUARE(x) ((x) * (x))
#define MAX(a, b) ((a) > (b) ? (a) : (b))

int y = SQUARE(5);       // y = 25
int m = MAX(10, 20);     // m = 20
```
Аргументы должны быть в скобках, чтобы избежать проблем с приоритетом операций. 

### Операторы \# и \##

\# — строкизация: превращает аргумент в строковый литерал.
```
#define STR(x) #x
printf(STR(hello));  // эквивалентно printf("hello");
```

Примеры использования 
Простая строкизация
```
#include <stdio.h>

#define STR(x) #x

int main() {
    printf("%s\n", STR(Hello World!));   // выведет: Hello World!
    printf("%s\n", STR(123));             // выведет: 123
    return 0;
}
```

Строкизация для отладки 
Часто используется для вывода имени переменной вместе с её значением: 
```
#define PRINT_INT(x) printf(#x " = %d\n", x)

int a = 42;
PRINT_INT(a);   // развернётся в: printf("a" " = %d\n", a);
                // после конкатенации строк: printf("a = %d\n", a);
```
Строкизация с escape-последовательностями 
Если аргумент содержит символы, требующие экранирования в строке (например, кавычки), препроцессор автоматически вставит обратную косую черту: 
```
#define ESC(x) #x
const char* s = ESC("quoted");   // развернётся в: "\"quoted\""
```
Формирование сообщений об ошибках 
Можно создавать макросы, которые выводят диагностические сообщения с именами параметров: 
```
#define ASSERT(cond) do { if (!(cond)) \
    printf("Assertion failed: %s, file %s, line %d\n", #cond, __FILE__, __LINE__); \
} while(0)
```
Вызов ASSERT(x > 0) выведет: Assertion failed: x > 0, file test.c, line 10


\## — склейка токенов.
```
#define VAR(name, num) name##num
int VAR(my_var, 1) = 10;  // объявляет переменную my_var1
```

Примеры использования 
Создание уникальных имён переменных 
Полезно, когда нужно объявить несколько похожих переменных: 
```
#define CREATE_VAR(name, num)  int name ## num

CREATE_VAR(myvar, 1) = 10;   // int myvar1 = 10;
CREATE_VAR(myvar, 2) = 20;   // int myvar2 = 20;
```
Генерирование функций для разных типов
```
#define DEFINE_MAX_FOR_TYPE(type) \
type max_##type(type a, type b) { \
    return a > b ? a : b; \
}

DEFINE_MAX_FOR_TYPE(int)    // создаёт функцию max_int
DEFINE_MAX_FOR_TYPE(double) // создаёт функцию max_double
```

### Отмена определения — #undef

Удаляет ранее определённый макрос.
```
#define TEMP 100
// ... код ...
#undef TEMP
#define TEMP 200   // можно переопределить
```

### Условная компиляция 

Позволяет компилировать или игнорировать части кода в зависимости от условий. Это мощный инструмент для создания кроссплатформенных программ, отладочных версий и т.д.  
Основные директивы: #if, #ifdef, #ifndef, #else, #elif, #endif. 

#ifdef / #ifndef — проверка определён ли макрос
```
#define DEBUG

#ifdef DEBUG
    printf("Отладочная информация\n");
#endif

#ifndef RELEASE
    printf("Это не релизная сборка\n");
#endif
```

#if — проверка константного выражения
```
#define VERSION 2

#if VERSION == 1
    // код для версии 1
#elif VERSION == 2
    // код для версии 2
#else
    // код для остальных версий
#endif
```

Заголовочные стражи (#ifndef + #define + #endif)
```
#ifndef MY_HEADER_H
#define MY_HEADER_H

// содержимое файла

#endif
```

### Директива #error — генерация ошибки

Если препроцессор встречает #error, компиляция прекращается и выводится указанное сообщение. Полезно для проверки обязательных условий. 
```
#ifndef __cplusplus
#error "Этот код требует компилятора C++!"
#endif
```

### Директива #pragma — инструкции компилятору
Это директива, специфичная для конкретного компилятора. Часто используется для управления оптимизациями, выравниванием структур, отключением предупреждений. 
```
#pragma pack(1)          // упаковка структур без выравнивания
#pragma warning(disable: 4996)  // отключить предупреждение (MSVC)
```

### Предопределённые макросы

Стандарт языка определяет несколько полезных макросов, которые всегда доступны:

\_\_LINE__ — текущая строка в файле.

\_\_FILE__ — имя текущего файла.

\_\_DATE__ — дата компиляции.

\_\_TIME__ — время компиляции.

\_\_STDC__ — равен 1, если компилятор соответствует стандарту ANSI C.

Пример использования:
```
printf("Ошибка в файле %s, строка %d\n", __FILE__, __LINE__);
```

## Порядок от младшего бита к старшему (на платформах Little-Endian)

Этот механизм используется на большинстве распространенных персональных компьютеров (архитектуры x86, x64). 
 
- Суть механизма: Компилятор размещает первое битовое поле, начиная с самого младшего бита (Least Significant Bit, LSB) 
первого байта выделенной области памяти. Заполнение идет в сторону увеличения номера бита внутри байта, а затем к следующим байтам. 
- Наглядное представление: Если представить байт как 8 бит, где биты нумеруются от 0 (младший) справа до 7 (старший) слева, 
то заполнение будет идти справа налево, а затем к следующему байту. 
 -Связь с порядком байт: Этот способ логически сочетается с little-endian представлением чисел, где младший байт числа хранится по меньшему адресу. 
 
Рассмотрим пример. Представьте структуру, которая должна уместиться в одно 32-битное слово:
```
struct Example {
    unsigned int a : 1; // Младший бит? (зависит от порядка)
    unsigned int b : 26;
    unsigned int c : 3;
};
```

В little-endian системе (при типичных настройках компилятора) распределение битов будет выглядеть так 

| Адрес   (смещение)     |	Биты в байте    |	Содержимое             |
|------------------------|------------------|------------------------|
| Байт 0 (младший адрес) |	7 6 5 4 3 2 1 0	| b[25:18] (часть поля b) |
| Байт 1                 | 7 6 5 4 3 2 1 0  |	b[17:10] (часть поля b) |
| Байт 2                 | 7 6 5 4 3 2 1 0  |	b[9:2] (часть поля b) |
| Байт 3  (старший адрес)|	7 6 5 4 3 2 1 0 | c[2:0] (поле c) b[1:0] a |


## Порядок от старшего бита к младшему (на платформах Big-Endian)

Этот механизм исторически использовался в сетевых протоколах (порядок байт "сетевой" — network byte order) и серверах (например, старых SPARC-машинах). 

- Суть механизма: Компилятор размещает первое битовое поле, начиная с самого старшего бита (Most Significant Bit, MSB) первого байта .
- Наглядное представление: Если использовать ту же нумерацию бит (7 — старший слева, 0 — младший справа), то заполнение идет слева направо.
- Связь с порядком байт: Этот способ логически сочетается с big-endian представлением чисел, где старший байт хранится по меньшему адресу.

  Для той же структуры Example в big-endian системе картина будет принципиально иной

| Адрес (смещение)       | Биты в байте    |	Содержимое                |
|------------------------|-----------------|----------------------------|
| Байт 0 (младший адрес) | 7 6 5 4 3 2 1 0 |	a b[25:19] (часть поля b) |
| Байт 1	               | 7 6 5 4 3 2 1 0 |	b[18:11] (часть поля b)   |
| Байт 2	               | 7 6 5 4 3 2 1 0 |	b[10:3] (часть поля b)    |
| Байт 3 (старший адрес) | 7 6 5 4 3 2 1 0 |	b[2:0] c[2:0] (поле c)    |

## Ключевые отличия и последствия для ваших задач
Эти два механизма напрямую ведут к проблемам, которые иллюстрируют ваши задачи:

 - Разница в именовании: В документации и обсуждениях вы можете встретить термины "left-to-right" (слева-направо) для big-endian
   и "right-to-left" (справа-налево) для little-endian . Это как раз про то, с какого конца байта начинается размещение.
 - Проблема переносимости: Если вы напишете код, который полагается на конкретный порядок бит (например, для структуры IPv4 из задачи 8),
   он может работать на вашем little-endian компьютере, но полностью сломается на big-endian системе, так как значения полей будут прочитаны из "перевернутых" бит .
 - Взаимодействие с сетевыми протоколами: Это критически важно. Сетевые протоколы, такие как IP, имеют фиксированную структуру и используют big-endian (сетевой порядок байт).
   При работе с ними  нельзя просто наложить битовое поле на буфер с пакетом. Порядок бит вашего компилятора (скорее всего little-endian) не совпадет с порядком бит в протоколе.
   Поэтому часто приходится либо использовать ручные битовые операции и макросы, либо применять специальные атрибуты компилятора (__attribute__((packed)), #pragma pack)
   и функции для преобразования порядка байт (htonl, ntohl), понимая, что переносимость все равно не гарантируется.

## Задача
Представление IPv4-адреса с помощью битовых полей 
Создайте структуру IPv4, которая представляет IP-адрес в виде 32‑битного целого числа, но позволяет обращаться к каждому октету отдельно через битовые поля. 
Используйте union для совмещения четырёх 8‑битных полей (octet1, octet2, octet3, octet4) и 32‑битного числа raw. 
Порядок полей должен соответствовать сете́вому (big‑endian), т.е. octet1 — старший байт. 

Напишите функции: 

- void setIP(IPv4& ip, uint8_t o1, uint8_t o2, uint8_t o3, uint8_t o4) – заполняет все октеты;
- void printIP(const IPv4& ip) – выводит адрес в точечно‑десятичной нотации (например, 192.168.1.1);
- uint32_t getRaw(const IPv4& ip) – возвращает целочисленное представление адреса.

В функции main() продемонстрируйте:

- создание адреса через setIP;
- вывод адреса;
- изменение одного октета через прямое присваивание полю структуры;
- повторный вывод и проверку, что изменилось и raw‑значение.

Ожидаемый результат: 
программа корректно отображает IP‑адрес и позволяет модифицировать отдельные октеты. 

## Задача

Битовые поля для регистров устройства (с union) 
Тема: битовые поля, union, отображение на память 
Уровень: продвинутый (для базового курса может быть как дополнительная) 

Представьте, что вы работаете с 8-битным регистром управления периферийным устройством. Биты регистра имеют следующее назначение:

- бит 0: enable (включение);
- биты 1–2: mode (2 бита, допустимые значения 0,1,2,3);
- бит 3: interrupt (разрешение прерывания);
- биты 4–7: зарезервированы (не используются, всегда читаются как 0).
 
Требуется определить структуру ControlRegister, которая с помощью битовых полей позволяет удобно обращаться к отдельным битам, 
но при этом занимает ровно 1 байт. Используйте union для совмещения битовых полей и целочисленного байта (uint8_t). Напишите функции:

- void setEnable(ControlRegister& reg, bool on);
- void setMode(ControlRegister& reg, int modeValue);
- bool isInterruptEnabled(const ControlRegister& reg);
- uint8_t getRaw(const ControlRegister& reg) – возвращает сырое значение регистра.
 
В main() продемонстрируйте установку параметров через битовые поля и проверку через сырое значение. 
 
Ожидаемый результат: структура-union позволяет обращаться как к отдельным полям, так и ко всему байту целиком.

Тема: условная компиляция, макросы 
Уровень: начальный

# Пространства имён (namespace) в C++

## Введение

При разработке крупных проектов на C++ неизбежно возникает проблема конфликта имён. Разные библиотеки, модули или даже части одной программы могут содержать функции, 
классы или переменные с одинаковыми именами. Компилятор не может различить их, что приводит к ошибкам. 
 
Пространства имён (namespace) — это механизм языка C++, предназначенный для логической группировки кода и изоляции имён. 
Они позволяют создавать контексты, внутри которых имена уникальны, и обращаться к ним через уточнение. 
Таким образом, одно и то же имя может существовать в разных пространствах имён без конфликтов. 

## Объявление пространства имён

Пространство имён объявляется с помощью ключевого слова namespace, за которым следует идентификатор и тело в фигурных скобках. 
Точка с запятой после закрывающей скобки не ставится.
```
namespace MyNamespace {
    int x = 10;
    void foo() {
        // ...
    }
    class MyClass {};
}
```
Всё, что объявлено внутри блока, принадлежит пространству имён MyNamespace

## Глобальное пространство имён

Любая программа на C++ имеет глобальное пространство имён. Имена, объявленные вне каких-либо пространств имён, находятся именно в нём. 
Обратиться к глобальному имени можно с помощью оператора :: без левой части:
```
int value = 5;  // глобальное пространство имён

namespace A {
    int value = 10;
    void print() {
        std::cout << ::value;   // выведет 5 (глобальное)
        std::cout << value;     // выведет 10 (A::value)
    }
}
```

## Доступ к членам пространства имён

Существует несколько способов использовать имена из пространства имён.

### Полностью квалифицированное имя

Имя указывается вместе с пространством имён через оператор разрешения области видимости ::
```
MyNamespace::foo();
int a = MyNamespace::x;
MyNamespace::MyClass obj;
```

###  Объявление using (using declaration)

Объявление using вводит конкретное имя из пространства имён в текущую область видимости. 
После этого имя можно использовать без уточнения.
```
using MyNamespace::foo;
foo();  // теперь можно вызывать без MyNamespace::
```
Объявление using действует только в той области видимости, где оно сделано (блок, функция, файл). 
Оно не вносит в область видимости другие имена из того же пространства. 

###  Директива using (using directive)

Директива using namespace делает все имена из указанного пространства имён доступными в текущей области видимости без уточнения.
```
using namespace MyNamespace;
foo();
x = 42;
MyClass obj;
```
Директива удобна, но может приводить к неоднозначности, если в текущей области уже есть имена, совпадающие с именами из импортированного пространства. 
Особенно опасна директива в заголовочных файлах, так как она навязывает импорт всем, кто включает этот заголовок. 
Поэтому рекомендуется использовать её осторожно, обычно только в небольших областях (например, внутри функции) или в файлах реализации. 

### Различия и рекомендации

- Полная квалификация — максимальная ясность, отсутствие побочных эффектов.
- using-объявление — удобно для часто используемых имён, сохраняет контроль над тем, что именно импортируется.
- using-директива — удобно для быстрого прототипирования, но может создавать конфликты. В профессиональном коде избегают в заголовках.

## Вложенные пространства имён и псевдонимы

Пространства имён могут быть вложенными.
```
namespace A {
    namespace B {
        namespace C {
            void func();
        }
    }
}
```

Обращение: A::B::C::func(); 
Начиная с C++17, можно использовать вложенное определение: 
```
namespace A::B::C {
    void func();
}
```

### Псевдонимы пространств имён

Для длинных имён можно создать псевдоним:
```
namespace abc = A::B::C;
abc::func();
```

##  Анонимные (безымянные) пространства имён

Анонимное пространство имён объявляется без имени. Его содержимое доступно только в пределах того файла, где оно объявлено, 
и при этом невидимо для других единиц трансляции.
```
namespace {
    int internalVar = 0;
    void internalHelper() {}
}
```
Это современная альтернатива использованию static для глобальных сущностей в C. В C++ анонимные пространства имён предпочтительнее, 
так как они могут содержать не только переменные и функции, но и типы.

##  Расширяемость пространств имён

Пространство имён можно дополнять в разных местах программы. Несколько блоков с одним именем объединяются компилятором.
```
// файл A.h
namespace MyLib {
    void funcA();
}

// файл B.h
namespace MyLib {
    void funcB();
}

// main.cpp
#include "A.h"
#include "B.h"
// теперь в MyLib доступны обе функции
```

Это свойство широко используется в стандартной библиотеке C++: всё содержимое заголовков находится в пространстве имён std

##  Примеры использования

Организация библиотеки
```
namespace Graphics {
    class Window {};
    void drawLine(Point p1, Point p2);
    namespace Effects {
        void blur(Image& img);
    }
}
```

Предотвращение конфликтов
```
namespace Library1 {
    void process(int);
}
namespace Library2 {
    void process(int);
}
// При необходимости уточняем: Library1::process(10);
```

Использование в одном файле
```
#include <iostream>

namespace MyCode {
    void hello() {
        std::cout << "Hello from MyCode\n";
    }
}

int main() {
    MyCode::hello();
    using MyCode::hello; // можно и так
    hello();
}
```

- Используйте пространства имён для логической группировки кода. Это улучшает читаемость и уменьшает вероятность конфликтов.
- Избегайте директив using namespace в заголовочных файлах. Они незаметно загрязняют глобальное пространство имён для всех, кто включает заголовок.
- Предпочитайте using-объявления или полную квалификацию. Это делает зависимости явными.
- Для внутренних сущностей файла используйте анонимные пространства имён вместо static.
- При разработке библиотек предусмотрите инлайн-пространства имён для версионирования.

  Пространства имён — мощный инструмент организации кода в C++. Правильное их использование способствует созданию чистых, масштабируемых и легко поддерживаемых проектов.
